# Meta-MAPG Neural Benchmarks

End-to-end implementation of the engineering plan in
`meta_mapg_neural_benchmarks_engineering_plan.md`: five learning-rule arms
(IPPO / Own-only / Peer-only / Full Meta-MAPG / Meta-MAPG warm-up → IPPO)
across **MPE**, **Overcooked-AI**, and **Melting Pot** with paired-seed,
LOLA-PG opponent-aware corrections, norm-clipped meta-gradients, and a
shape-then-cool handoff experiment.

## TL;DR — single command

```bash
unzip meta_mapg_neural.zip
cd meta_mapg_neural
bash setup.sh                       # uv venv + deps + GPU verify

# 5-minute pre-flight smoke — ALWAYS run this first before the 24h job:
PRE_FLIGHT=1 bash run_all.sh        # 2 seeds × MPE only × no pilot/λ
                                    # confirms CUDA + multiproc + figure
                                    # pipeline are alive end-to-end.

# Full sweep: 24-32h on 4×V100, kicked off detached.
nohup bash run_all.sh > run_all.log 2>&1 &
tail -f run_all.log

# Stop a running sweep (and all its children).
bash kill.sh --status      # just show what's running
bash kill.sh               # SIGTERM, then SIGKILL stragglers after 5s
bash kill.sh --force       # immediate SIGKILL
```

After the sweep completes, figures land in `artifacts/figures/` and a
timestamped self-contained zip (`meta_mapg_neural_results_*.zip`) is written
to the repo root.

## What's implemented

### Five arms (engineering plan §2)

| Arm | `lam_peer` | `lam_own` | Notes |
|----|----:|----:|-----|
| `ippo`      | 0.0 | 0.0 | independent PPO baseline |
| `own_only`  | 0.0 | 0.5 | Meta-PG without peer term |
| `peer_only` | 1.0 | 0.0 | LOLA-style opponent-aware (cleanest mechanism test) |
| `meta_mapg` | 1.0 | 0.5 | full Meta-MAPG = own + peer |
| `handoff`   | 1.0 → 0.0 at `T_warm` | 0.5 → 0.0 | the headline experiment §6.5 |

### Correctness

The opponent-aware correction is the **score-function LOLA-PG** form:

```
J_i^score = E[ A_i · ( log π_i(a_i) + Σ_{j≠i} log π_j(a_j) ) ]
Δ_peer^i = η · ∇_{θ_i} [ (∇_{θ_j} J_j)^⊤ (∇_{θ_j} J_i^score) ]
Δ_own^i  = η · ∇_{θ_i} ‖ ∇_{θ_i} J_i ‖²
```

Implemented with `torch.autograd.grad(..., create_graph=True)`.  Unit tests in
`tests/test_corrections.py` verify this is *not* the placeholder
`(g_peer · θ_peer)` form previously seen in this repo.

### Engineering-plan compliance

* §4 — correction applied to **actor only**;  critic uses plain MSE.
* §4 — `advantages.detach()` everywhere on the meta path (stop-grad on values).
* §4 — `‖Δ_corr‖ ≤ c · ‖Δ_PG‖` enforced via `clip_correction()` (default c=1.0).
* §4 — short unrolls only (one-step LOLA).
* §8 — paired seeds: `init_paired_actors(seed=…)` produces deterministic,
  identical initial weights across all arms.
* §8 — bootstrap CIs / Wilson intervals for success fractions.

### Benchmarks

* **MPE** (PettingZoo): `simple_spread`, `simple_reference`, `simple_speaker_listener`.
* **Overcooked-AI** (Stanford HARL): `cramped_room`, `coordination_ring`, `forced_coordination`.
* **Melting Pot** (`dm_meltingpot`): `prisoners_dilemma_in_the_matrix__repeated`, `clean_up`.
  *Falls back to a PettingZoo-style iterated prisoner's dilemma if `dm_meltingpot`
  fails to install (its native deps are heavy on x86_64 Ubuntu — see plan §7.2).*

## Reproducing the paper figures

```bash
bash setup.sh                                       # env
nohup bash run_all.sh > run_all.log 2>&1 &          # full pipeline (24+ h on 4×V100)
ls artifacts/figures                                 # outputs
```

To run a smoke test in ≤ 30 minutes:

```bash
SEEDS_OVERRIDE=4 SKIP_OVERCOOKED=1 SKIP_MELTINGPOT=1 \
SKIP_LAMBDA=1 bash run_all.sh
```

5-minute pre-flight (always do this first on a new box):

```bash
PRE_FLIGHT=1 bash run_all.sh        # 2 seeds × MPE only, no pilot/λ
```

**Pilot → threshold → real-run flow.** When `SKIP_PILOT` is *not* set, every
benchmark first runs IPPO with 4 seeds, fits a 75th-percentile-of-final-return
threshold, and writes it to `artifacts/pilot/{benchmark}/{env_id}/threshold.txt`.
The full sweep then picks this up automatically via `--pilot-dir
artifacts/pilot`.  This is the engineering plan §5.4 / §6.4 / §7.4 protocol —
do NOT skip it on a fresh box, or your basin-entry threshold may saturate at
0% or 100% and the figures will be useless.

To run **just one benchmark**:

```bash
python scripts/pilot.py    configs/mpe_simple_spread.yaml
# (inspect the suggested threshold; edit configs/mpe_simple_spread.yaml: threshold)
python scripts/run_sweep.py configs/mpe_simple_spread.yaml --max-parallel 4
python scripts/make_figures.py
```

## Hardware expectations

* **Designed for**: 4 × NVIDIA V100-SXM2-32GB (or any 4-GPU box, CUDA 12.x).
* **Single GPU**: works but reduces parallelism — set `--max-parallel 1`.
* **CPU-only**: works for MPE smoke tests; not realistic for full sweep.

Estimated wall time (4 × V100):

| Stage | Wall time |
|----|----:|
| MPE pilot + 3 envs × 30 seeds × 1 M steps | ~3 h |
| Overcooked 3 layouts × 30 seeds × 2 M steps | ~12-18 h |
| Melting Pot 2 substrates × 30 seeds × 1.5 M steps | ~6-10 h |
| λ-sweep on `simple_spread` (8 lambdas × 12 seeds) | ~2 h |
| **total** | **24-32 h** |

Use `nohup` so the pipeline survives Jupyter session disconnects.

## Statistical protocol (plan §8)

* `init_paired_actors(seed=s)` produces identical initial weights across arms;
  the same seed is used for every arm in the sweep.
* Success-fraction CIs are Wilson 95% intervals.
* Return CIs are bootstrap 95% (2000 resamples).
* All eval is done with **deterministic argmax** rollouts (50 episodes/checkpoint).

## Plan §10 — what we are NOT doing

* No leaderboard benchmarking against every MARL baseline.
* No peak-return optimisation.
* No silent failures: if a benchmark shows no gain, it's reported as such.
* No reliance on a single lucky seed.

## Known limitations

* **Melting Pot install** (`dm_meltingpot`) often fails on x86_64 without
  bazel + dmlab2d build infrastructure.  In that case the IPD fallback runs,
  which the plan permits as a "matrix-style social dilemma" §7.2.
* **Overcooked-AI** depends on `pygame` and renders during init; on headless
  servers set `SDL_VIDEODRIVER=dummy` (already done by `run_all.sh` if needed).
* The threshold in each config defaults to the value cited in the plan.  Plan
  §5.4 / §6.4 require running a pilot first; `scripts/pilot.py` prints the
  recommended 75-th-percentile threshold.
